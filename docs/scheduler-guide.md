# 调度组配置指南

调度组可以先按一个简单模型理解: 它是一层本地路由器. 请求进来以后, 调度组决定下一步去哪个上游, 或者进入哪个调度组继续判断.

最常见的配置是一个入口调度组负责分流: 普通模型走默认上游, GLM 模型走国产中转, 生图模型走独立生图上游. 下面按这个场景从头配置一遍.

## 先准备上游

先在上游页面添加真实服务. 比如你可能会有三个上游:

- 普通 OpenAI 兼容上游, 用来跑默认模型
- 国产 OpenAI 中转, 用来跑 `glm-*`
- 生图中转, 用来跑 `gpt-image-*`

上游只代表真实服务本身. 哪些模型走哪个上游, 后面交给调度组决定.

## 建几个能力组

建议先建能力组, 每个组只负责一类能力.

比如建一个 `默认模型` 调度组. 模式可以选失败切换, 然后把普通上游放进组内目标. 这样默认模型就有自己的主备策略.

再建一个 `GLM 中转` 调度组. 如果你只有一个国产中转, 模式选固定就够了. 如果有多个国产中转, 可以选失败切换.

生图也可以单独建一个 `Image` 调度组, 里面只放生图上游.

这一步的目的不是分流模型, 而是把每类能力先整理成一个可复用的目标. 后面入口组只负责把请求分到这些能力组.

## 建入口模型映射组

新建一个调度组, 比如叫 `入口路由`, 模式选择模型映射.

模型映射和固定, 随机, 轮询, 失败切换是平级模式. 只有调度组模式是模型映射时, 模型路由规则才会生效.

在 `入口路由` 里添加几条规则:

| 名称 | Pattern | 目标类型 | 目标 |
| --- | --- | --- | --- |
| GLM | `glm-*` | 调度组 | `GLM 中转` |
| Image | `gpt-image-*` | 调度组 | `Image` |
| Default | `*` | 调度组 | `默认模型` |

默认规则 `Pattern = *` 会匹配所有模型, 所以它应该放在最后兜底. 实际配置时, 给 GLM 和 Image 更高优先级, 给 Default 更低优先级.

最后在调度组列表里选中 `入口路由`, 让请求都先进入这个入口组.

新建调度组保存成功后会自动成为当前入口. 内置 `Default` 组不能删除. 删除当前自建组后, 应用会回退到 `Default`.

## 请求会怎么走

配置完成后, 可以这样理解路由过程:

```text
glm-4.5       -> 入口路由 -> GLM 中转 -> 国产中转上游
gpt-image-1  -> 入口路由 -> Image -> 生图上游
gpt-5        -> 入口路由 -> 默认模型 -> 普通上游
```

如果目标是上游, 请求会直接转发到上游. 如果目标是调度组, 请求会进入那个调度组继续解析.

## 普通模式怎么选

模型映射适合做入口分流. 如果某个调度组不需要按模型名判断, 就使用普通模式.

| 模式 | 适合场景 |
| --- | --- |
| 固定 | 锁定一个上游或一个调度组 |
| 随机 | 按上游权重随机选择一个目标, 不自动重试 |
| 轮询 | 按上游权重轮流选择一个目标, 不自动重试 |
| 失败切换 | 按优先顺序尝试主备上游, 可在失败后切到备用 |

普通模式下, 关闭 `使用全部目标` 后可以同时选择上游和子调度组. 随机, 轮询和失败切换会把子调度组中的可用上游展开到父组的候选池, 然后只执行父组策略. 子调度组自身的随机, 轮询或失败切换模式不会执行.

如果需要真正进入子调度组并继续执行它的模式, 使用 `固定 -> 调度组` 或 `模型映射 -> 调度组`.

## 会话亲和

请求 JSON 中存在 `prompt_cache_key`, `conversation_id` 或 `session_id` 时, 成功请求会在亲和 TTL 内优先复用同一上游.

- 随机和轮询在 TTL 内只选择亲和上游.
- 失败切换把亲和上游移到首位, 但仍保留备用候选.
- 上游禁用或不支持当前 endpoint 时, 已有亲和会被忽略.
- 亲和, 轮询游标和失败计数都只保存在内存中, 应用重启后清空.

亲和 key 还包含调度组, endpoint 和有效模型, 因此不同模型或 endpoint 不会误用同一条亲和记录. TTL 最低为 `60` 秒.

## 失败切换细节

失败切换可以分别开启余额不足, 网络失败和 5xx 的立即切换. `失败阈值` 表示同一上游累计多少次失败后进入切换逻辑.

即使某个立即切换开关关闭, 累计失败达到阈值后仍可能切换. `402`, `429`, `5xx`, 网络错误和常见余额不足响应会参与失败判断. 普通 `4xx` 通常不会自动切换.

上游成功一次后会清除它在当前组中的失败计数. 达到阈值的上游不会永久禁用, 只会排在健康候选之后. 所有候选都不健康时仍会继续尝试.

## 名称和 Pattern

名称只是备注, 给人看, 不参与匹配. 可以写 `GLM`, `生图`, `默认兜底` 这类容易认的名字.

Pattern 才是真正用来匹配请求模型名的字段.

启用规则按优先级从高到低排序. 优先级相同时, Pattern 较长的规则先匹配, 再按创建时间排序. 第一条命中后就停止, 匹配区分大小写.

| Pattern | 含义 |
| --- | --- |
| `glm-4.5` | 只匹配 `glm-4.5` |
| `glm-*` | 匹配 `glm-4`, `glm-4.5`, `glm-z1` |
| `gpt-image-*` | 匹配一组生图模型 |
| `claude-3-?` | `?` 匹配一个字符 |
| `*` | 匹配所有模型 |

## 目标模型改写

目标模型为空时, 不再改写当前有效 `model`. 如果此前没有任何规则改写模型, 上游会收到客户端模型. 在嵌套路由中, 空值会沿用前一层已经改写的模型.

如果上游模型名和客户端模型名不同, 就填写目标模型. 比如客户端用 `my-image`, 但上游需要 `gpt-image-1`, 那就把目标模型填成 `gpt-image-1`.

目标模型也支持通配捕获. Pattern 里的 `*` 和 `?` 会捕获内容, 目标模型里的 `*` 和 `?` 会按顺序替换.

| 配置项 | 值 |
| --- | --- |
| Pattern | `glm/*` |
| 客户端模型 | `glm/glm-4.5` |
| 目标模型 | `*` |
| 上游收到 | `glm-4.5` |

再看一个多段捕获:

| 配置项 | 值 |
| --- | --- |
| Pattern | `vendor/*/model-?` |
| 客户端模型 | `vendor/acme/model-a` |
| 目标模型 | `*/?` |
| 上游收到 | `acme/a` |

## 最大跳转次数

最大跳转次数是全局路由保护配置. 固定模式跳到调度组, 或模型映射命中调度组或上游时, 都会算一次跳转. 普通模式展开子调度组也受这个上限保护.

这个配置主要用来防止固定或模型映射真实跳转链中的环形路由. 比如 A 通过固定或模型映射跳到 B, B 又跳回 A, 跳转最终会达到上限.

随机, 轮询和失败切换中的普通子组关系会展开为上游池并按组去重, 不会进入子组执行自身模式. 这类子组互引不应按真实跳转环路排查.

一般保持默认值就够用. 只有调度组嵌套很多层时, 才需要调大.

## 常见问题

如果模型映射没有生效, 先确认当前调度组是不是入口模型映射组, 再确认这个组的模式是否真的是模型映射.

如果请求打到了错误上游, 检查规则优先级. 默认规则 `*` 的优先级应该低于更具体的规则.

如果上游收到的模型名不对, 检查目标模型. 目标模型为空表示不再改写当前有效模型, 填了固定值就会从该层开始改写.

如果出现最大跳转次数错误, 检查固定或模型映射规则中是否有 A -> B -> A 这样的真实跳转路径, 或者路由层级是否超过上限.

如果随机或轮询看起来一直选择同一上游, 检查请求是否带有稳定的 `prompt_cache_key`, `conversation_id` 或 `session_id`. 在亲和 TTL 内, 这种行为是预期结果.

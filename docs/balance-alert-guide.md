# 余额提醒指南

余额模块可以手动查询 Relay API Key 上游的余额, 也可以定时检查并在余额不足时发送系统通知.

## 先验证余额查询

1. 打开 `上游`.
2. 找到一个 Relay API Key 上游并点击 `编辑`.
3. 选择正确的 `余额 provider`.
4. 显式选择 NewApi, 或 auto 识别为 NewApi 时, 用户 Key 和用户 ID 都必须填写.
5. 保存后在上游列表点击 `查余额`.

只有手动查询成功后, 才建议开启系统提醒. 手动点击 `查余额` 只刷新余额快照, 不会立即执行提醒判断.

支持的 provider 及凭据说明见[上游管理指南](upstream-guide.md).

## 开启提醒

在上游编辑器的 `余额不足提醒` 区域配置:

| 字段 | 默认值 | 说明 |
| --- | --- | --- |
| 启用系统提醒 | 关闭 | 是否加入后台定时检查 |
| 余额阈值 | `5.0` | `remaining <= 阈值` 时视为余额不足 |
| 检查间隔秒 | `1800` | 同一上游两次检查的最短间隔, 最低 `60` 秒 |

阈值直接使用余额快照中的单位, 不进行通用汇率换算. 如果 provider 返回 CNY, 阈值就是 CNY. 返回 USD 时, 阈值就是 USD.

提醒只支持已启用的 Relay API Key 上游. Codex OAuth 不会执行余额提醒. `unsupported` provider 不应启用提醒, 因为它无法查询或触发通知, 但已启用配置仍会按周期尝试并记录失败.

## 后台检查

余额提醒任务随桌面应用启动, 每 `30` 秒扫描一次到期配置. 应用可以隐藏到托盘, 但进程必须保持运行.

每次到期检查会请求 provider. 查询成功并得到快照时才会覆盖持久化余额. 缺凭据, 不支持 provider 或传输错误不会覆盖旧快照, 但仍会更新本轮提醒检查时间. 因此界面可能继续显示旧余额.

成功得到快照后, 提醒状态按以下规则变化:

- 首次发现余额小于等于阈值时发送系统通知.
- 持续处于低余额状态时不重复通知.
- 余额恢复到阈值以上时, 静默清除提醒状态.
- 恢复后再次跌破阈值时, 才发送下一次通知.

通知发送失败时不会把提醒标记为已发送, 下一个完整检查周期会重试. 查询接口报错时会记录检查时间并保留原提醒状态.

## 修改设置后的行为

修改启用状态, 阈值或检查间隔会重置最近检查时间和活动提醒状态. 保存后通常会在下一次 `30` 秒扫描中重新查询. 如果余额仍然不足, 可能再次发送通知.

提醒状态持久化在 SQLite. 仅重启应用不会无条件重复发送同一个低余额通知.

## 页面状态

上游列表会显示提醒是否开启. 低余额提醒处于活动状态时, 列表会显示红色的 `不足 <= 阈值`.

系统通知可能受到操作系统通知权限, 勿扰模式或桌面环境支持情况影响. 当前不支持邮件, Webhook 或自定义通知渠道.

## 特殊情况

提醒判断只读取快照中的 `remaining`:

- `remaining` 有值时, 使用它与阈值比较.
- `remaining` 为 `None` 时, 本次结果视为未低于阈值.
- 如果此前提醒处于活动状态, 一次成功但没有 `remaining` 的响应会清除活动状态.
- 查询直接报错时不会清除活动状态.

遇到上游列表的余额列显示未知或失败时, 应先修正 provider 和凭据, 不要依赖提醒状态判断真实余额.
